# 后续开发方案：个人稳定版到开源协作预备

## 1. 当前基线

项目当前处于 MVP 便携体验版：

- 支持早上计划、白天执行、晚上复盘。
- 支持文件夹视图和星系视图切换。
- 支持任务四象限、开始/完成、实际耗时、顺延原因。
- 支持便携 zip 打包，用户数据和示例数据分离。
- 数据本地保存到便携目录内的 SQLite。

后续开发不应推翻这条主线。短期目标是让个人长期使用稳定、可备份、可恢复、可发给别人体验；中期目标是改善复盘洞察和月度视图；长期目标才是开源协作和小团队可能性。

## 2. 产品原则

- 本地优先：默认不需要账号、不需要联网、不依赖云服务。
- 个人优先：先服务一个人每天真实使用，不提前做团队协作。
- 低压力反馈：展示完成率、趋势、分布和变化，不做强刺激分数排名。
- 数据可控：用户必须知道数据在哪里、如何备份、如何恢复、如何导出。
- 渐进发布：每个阶段都能独立交付，不把多个大目标捆在一起。

## 3. 能力边界

当前和下一阶段明确不做：

- 不做账号系统。
- 不做云同步。
- 不做多人实时协作。
- 不做复杂权限模型。
- 不做移动端。
- 不做浏览器 SaaS 版本。
- 不做 AI 自动评价用户表现。
- 不做强制打分、排行榜或绩效化评分。

当前能力限制：

- 便携版依赖目标电脑已有 Microsoft Edge WebView2 Runtime。
- SQLite 数据没有加密，不能保存敏感隐私、密码、财务凭据等高风险内容。
- 便携 zip 不包含源码，但 exe 仍然可能被逆向分析。
- 当前月度总览是轻量能力，不是完整统计系统。
- 当前提醒能力不应被视为系统级日历或任务管理器替代品。

只有在个人稳定版经过真实使用后，才考虑：

- 数据加密。
- 云同步。
- 团队协作。
- 跨设备同步。
- 标准安装版自动更新。

## 4. 测试边界

每个后续阶段都至少覆盖以下测试层级：

- 领域规则单元测试：任务状态、顺延、复盘、趋势计算、导入导出规则。
- 数据仓库测试：Memory repository 和 Tauri SQLite repository 的一致性。
- UI 组件测试：首屏入口、任务操作、复盘表单、月度视图、导入导出弹窗。
- 打包测试：便携 zip 内容、运行时 DLL、快速开始文档、启动 smoke。
- 文档检查：README、快速开始、阶段方案、实施计划不能互相矛盾。

不承诺覆盖的测试范围：

- 不做完整 Windows 版本矩阵测试。
- 不做 macOS/Linux 发布测试，除非对应阶段明确纳入。
- 不做高并发或服务器压力测试，因为当前没有服务器。
- 不做正式安全审计。
- 不做所有杀毒软件误报兼容性验证。

手动验收必须保留，因为桌面 GUI 和打包结果无法完全用单元测试替代。

## 5. 阶段路线

### Phase 3：数据安全和可恢复性

目标：让个人长期使用时不怕数据丢失，并能放心把体验包发给别人。

交付内容：

- 数据导出：导出 JSON，保留完整数据结构。
- 数据导入：从 JSON 恢复，导入前做格式校验。
- 人类可读导出：导出 Markdown 或 CSV，用于备份和回顾。
- 应用内备份入口：显示当前数据位置，提供打开数据文件夹和备份说明。
- 示例数据重置：一键重建 demo 数据，不影响我的数据。
- 数据迁移说明：解释从旧便携目录迁移到新目录的步骤。

能力边界：

- 不做云备份。
- 不做自动后台备份。
- 不做加密备份。
- 不承诺导入第三方待办软件格式。

验收标准：

- 导出的 JSON 能完整恢复任务、每日文件、复盘、设置。
- 导入非法 JSON 时不破坏现有数据。
- 用户能在 UI 中找到数据位置和备份说明。
- demo 重置不会修改 user 数据库。

### Phase 4：复盘洞察和月度体验

目标：把“完成率和趋势展示”做实，同时避免数字绑架。

交付内容：

- 月度完成趋势：按日期展示完成率、任务数量、重要任务完成情况。
- 四象限分布：展示任务落在哪些象限，以及完成变化。
- 顺延原因趋势：统计常见顺延原因和备注。
- 实际耗时趋势：展示计划和执行之间的偏差，不给人格化评价。
- 星系月度地图：30/31 个节点展示每日状态，点击进入当日任务。
- 文件柜月度视图：按日格子查看每日任务文件。

能力边界：

- 不做绩效评分。
- 不做排名。
- 不用算法给用户贴标签。
- 不把效率简化成单一分数。

验收标准：

- 趋势图能处理空数据、部分数据、跨月数据。
- 月度视图点击任意日期能查看当日计划和复盘。
- 星系和文件柜月度视图共享同一数据来源。
- 所有趋势文案保持中性，不制造压力。

### Phase 5：日常工作流增强

目标：减少每天重复输入，提高真实使用效率。

交付内容：

- 常用任务模板：用户可以保存和复用任务组合。
- 轻量重复任务：支持每天、工作日、每周的简单重复。
- 计划耗时和实际耗时对比。
- 搜索和筛选：按标题、日期、象限、状态、顺延原因查找。
- 快捷操作：常用操作支持键盘或更少点击。
- 复盘草稿保护：避免误关闭导致输入丢失。

能力边界：

- 不做复杂周期规则，例如每月倒数第二个工作日。
- 不做跨人任务分配。
- 不做项目管理甘特图。
- 不做完整日历系统。

验收标准：

- 模板不会自动污染今天任务，必须由用户明确添加。
- 重复任务生成有可解释规则，且可关闭。
- 搜索在没有结果时给出清晰空状态。
- 草稿保护不会阻碍用户主动清空。

### Phase 6：开源发布和协作准备

目标：让 GitHub 仓库对外部用户和贡献者可理解、可运行、可反馈。

交付内容：

- 完整 README：项目介绍、截图、安装、开发、打包、数据说明。
- CONTRIBUTING：如何提交 issue、如何运行测试、代码风格。
- CHANGELOG：记录版本变化。
- LICENSE：明确开源协议。
- GitHub issue 模板：bug、功能建议、体验反馈。
- GitHub Actions：至少跑前端测试、构建检查、Tauri check。
- Release 草案：上传便携 zip 或说明本地生成方式。

能力边界：

- 不承诺马上维护外部 PR。
- 不承诺跨平台二进制包。
- 不承诺自动更新。
- 不接受涉及隐私数据的 issue 附件。

验收标准：

- 新用户能按 README 在新机器上跑起来。
- CI 能在提交时发现基础测试失败。
- Release 页面能说明 WebView2 Runtime 前置条件。
- issue 模板引导用户提供系统、版本、复现步骤。

### Phase 7：小团队可能性验证

目标：只验证团队场景是否值得做，不直接做完整团队系统。

交付内容：

- 团队需求调研文档：明确谁用、协作什么、共享哪些数据。
- 数据边界方案：个人数据和共享数据如何分开。
- 只读分享导出：先用文件导出分享，不做实时同步。
- 小团队原型评估：判断是否需要账号、权限、云同步。

能力边界：

- 不做实时多人编辑。
- 不做组织架构。
- 不做管理员后台。
- 不做企业级权限。
- 不做收费系统。

验收标准：

- 能清楚回答团队功能是否真的必要。
- 能列出团队化会带来的隐私、数据一致性、成本和维护负担。
- 如果继续做团队版，必须先写新的独立方案和计划。

## 6. 决策门槛

进入下一阶段前必须满足：

- 当前阶段关键验收标准通过。
- README 和对应方案文档已更新。
- 关键风险已经写入能力边界或已处理。
- 没有未解释的数据迁移风险。
- 至少完成一次手动 smoke 测试。

任何涉及云同步、账号、加密、团队协作的改动，都需要重新写独立方案，不应作为顺手功能加入。

## 7. 推荐优先级

优先执行顺序：

1. Phase 3 数据安全和可恢复性。
2. Phase 4 复盘洞察和月度体验。
3. Phase 5 日常工作流增强。
4. Phase 6 开源发布和协作准备。
5. Phase 7 小团队可能性验证。

原因：数据安全是个人长期使用的基础；洞察和月度体验是产品差异化；工作流增强提高留存；开源发布需要前面能力稳定；团队化不应早于个人版稳定。
