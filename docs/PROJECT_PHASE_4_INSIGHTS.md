# 四阶段方案：复盘洞察和月度体验

## 阶段目标

把每日计划和晚上复盘沉淀成可查看的月度趋势，让用户能看到任务完成情况、重要任务推进、四象限分布、顺延原因和实际耗时变化。

本阶段坚持低压力反馈：展示完成率和趋势，不做分数、排名、绩效评价或人格化判断。

## 本阶段包含

- 月度完成趋势：按日期展示任务数量、完成数量和完成率。
- 重要任务完成情况：把重要任务的完成情况从全部任务里单独拆出来。
- 四象限分布：展示任务在四象限中的数量和完成变化。
- 顺延原因汇总：统计常见顺延原因和是否填写备注，用于复盘输入。
- 实际耗时趋势：按日期汇总已记录任务的实际耗时。
- 月度星系地图：用 30 或 31 个节点表达当月每日状态，点击节点查看当日任务。
- 月度文件柜视图：用每日格子查看当日任务文件，不替代今日工作台。

## 本阶段不包含

- 不做效率分数。
- 不做排行榜或与他人比较。
- 不用算法评价用户表现。
- 不把完成率解释成好坏判断。
- 不做云同步、账号、团队报表或跨设备统计。
- 不引入复杂图表依赖，除非已有 CSS/SVG 实现明显不可维护。

## 指标口径

### 完成率

完成率只表示任务状态中 `completed` 的占比：

```text
完成率 = 已完成任务数 / 当日任务总数
```

没有任务的日期不参与趋势计算，也不显示为 0 分。

### 重要任务完成情况

重要任务包括：

- 重要且紧急。
- 重要不紧急。

重要任务完成率单独展示，不合并成总分。它用于提醒注意力分布，不用于评价当天表现。

### 四象限分布

四象限展示任务落点和完成状态。它回答的是“任务结构是什么样”，不是“做得好不好”。

### 顺延原因

顺延原因按原因标签统计，同时记录是否填写了说明。原因说明用于帮助用户发现阻塞模式，不用于标记失败。

### 实际耗时

实际耗时按任务会话记录汇总。用户允许事后修改耗时，因此趋势只反映当前记录值，不推断真实生产力。

## 文案规则

允许使用：

- 完成率。
- 趋势。
- 分布。
- 推进。
- 顺延原因。
- 实际耗时。
- 本月记录。

避免使用：

- 失败。
- 低效。
- 落后。
- 扣分。
- 表现差。
- 拖延症。
- 不合格。

## UI 原则

- 月度总览从今日工作台的按钮进入，不替代主页。
- 今日主页继续保留文件夹视图和星系视图切换。
- 月度趋势优先可读性，动画只作为辅助趣味。
- 空数据状态要中性，说明“本月还没有任务记录”，不要给压力。
- 颜色用于区分状态和象限，不表达惩罚。

## 测试边界

自动测试至少覆盖：

- 空月份返回空趋势。
- 部分完成日期能计算完成率。
- 重要任务完成情况和全部任务完成情况分开计算。
- 顺延原因按标签统计，并区分是否填写备注。
- 实际耗时按日期聚合。
- 月度总览显示完成趋势、四象限分布、顺延原因和空状态。

手动验收至少覆盖：

1. 没有任何任务记录的月份。
2. 只有部分日期有记录的月份。
3. 有顺延任务和顺延原因的月份。
4. 月度星系节点点击后能查看当日任务。
5. 月度文件柜格子点击后能查看当日任务。
6. 页面文案保持中性，不出现打分或评价式语言。

## 验收标准

- 趋势计算是纯领域逻辑，不依赖 React 或 SQLite。
- 月度总览能处理空数据、部分数据和跨月数据。
- 重要任务完成情况不会被折算成单一效率分数。
- 顺延原因被作为复盘材料展示，不被描述为失败。
- 星系月度地图和文件柜月度视图共享同一份月度数据。
- README 和本方案对能力边界的描述一致。
